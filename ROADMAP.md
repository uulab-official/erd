- [x] Canvas Entity 노드/Relationship 엣지 실제 렌더링 — `packages/canvas`의 `EntityNode`(로지컬/피지컬 이름, PK/FK 뱃지가 붙은 attribute 목록, 4방향 Handle)가 기존 "라벨만 있는 플레이스홀더 박스"를 대체. 엣지는 `Relationship.kind`에 따라 실선(identifying)/점선(non-identifying), 화살표 마커, `name (cardinality)` 라벨로 스타일링 — [PR #20](https://github.com/uulab-official/erd/pull/20)
- [x] 캔버스 드래그로 Relationship 생성 — 한 Entity의 Handle을 다른 Entity로 드래그(React Flow `onConnect`)하면 `connectEntitiesCascade`가 대상 Entity에 소스의 PK를 미러링하는 FK Attribute를 만들고(`AddAttribute`) 그 다음 `CreateRelationship`을 실행, 하나의 Transaction으로 원자적 Undo — [PR #21](https://github.com/uulab-official/erd/pull/21). 소스가 PK 없음/복합 PK인 경우는 에러를 던지고 `Workspace.tsx`가 `window.alert`로 보여준다(전용 토스트 UI는 아직 없음). 기본값은 `cardinality: "one-to-many"`, `kind: "non-identifying"`, `optionality: "mandatory"`.
- [x] Subject Area 관리 (erwin식 캔버스 그룹핑) — `SubjectArea` 타입과 `Entity.subjectAreaId`는 schema-engine에 이미 있었지만(Domain/Dictionary처럼) Operation도 UI도 없었다. `Model.subjectAreas?`(Domain과 동일한 optional 패턴) + `CreateSubjectArea`/`UpdateSubjectArea`/`DeleteSubjectArea`/`AssignEntityToSubjectArea`/`UnassignEntityFromSubjectArea` Operation, `deleteSubjectAreaCascade`(멤버 전체 unassign 후 삭제, 원자적 Undo). `SubjectArea.entityIds`와 `Entity.subjectAreaId`는 Attribute가 최대 하나의 Domain에만 속하는 것과 같은 단일-소속 모델이고, 이 두 필드는 오직 이 다섯 Operation을 통해서만 함께 갱신되어 서로 어긋나지 않는다. 캔버스에는 `SubjectAreaNode`가 멤버 Entity들의 현재 위치를 감싸는 배경 박스(점선 테두리+색상 틴트+라벨)를 그린다 — 드래그 불가/선택 불가/연결 불가로 순수 시각 요소이며 `zIndex: -1`로 항상 Entity/엣지 뒤에 위치, 멤버가 이동하면 박스도 같이 재계산된다. GovernancePanel에 새 "Subject Areas" 탭 — [PR #29](https://github.com/uulab-official/erd/pull/29)
- [x] Relationship Inspector — 캔버스에서 엣지를 클릭하면 우측 사이드패널이 열려 이름/cardinality/kind/optionality/onDelete/onUpdate 편집과 Relationship 삭제가 가능 — [PR #24](https://github.com/uulab-official/erd/pull/24). 드래그 생성 기본값(1:N non-identifying)을 UI에서 바꿀 수 없던 구멍을 막았다. 이름·optionality·FK 액션은 새 `SetRelationshipMeta` Operation(`SetEntityMeta`와 같은 부분-메타 패턴)으로 바꾼다.
- [x] Auto Layout — `packages/layout-engine`에 관계 기반 `layeredLayoutEngine` 추가(의존성 없는 Sugiyama-lite: longest-path로 레이어 배정 → 부모 barycenter로 레이어 내 정렬, 부모는 왼쪽·자식은 오른쪽, 고립 엔티티는 마지막 별도 컬럼, 순환 관계에서도 종료 보장). Toolbar의 "Layout…" 셀렉트에서 실행하면 `moveEntitiesTransaction`(erd-engine의 새 Transaction 헬퍼)이 바뀌는 엔티티에만 `MoveEntity`를 묶어 **한 번의 Undo로 전체 배치가 복원**된다. ELK/Dagre 수준의 교차 최소화는 하지 않는다 — ERD는 수십 개 노드 규모라 단순한 알고리즘이 유지보수에 낫고, 필요해지면 같은 `LayoutEngine` 인터페이스로 ELK 백엔드를 추가하면 된다 — [PR #26](https://github.com/uulab-official/erd/pull/26)
- [x] 캔버스에서 Entity 드래그 이동 — 그동안 `MoveEntity` Operation만 있고 캔버스가 controlled `nodes`에 `onNodesChange`조차 없어서 **노드를 아예 움직일 수 없었다**. `ErdCanvas`가 semi-controlled로 전환: 드래그 중에는 로컬 React Flow 상태로만 움직이고(`applyNodeChanges`, mousemove마다 Operation을 쓰지 않음), 드래그가 끝나면 `onMoveEntity`로 최종 좌표 1건만 보고 → `MoveEntity` Operation으로 기록(Undo/Redo/Save 전부 적용). Model이 바뀌면(undo 포함) `useEffect`가 노드 위치를 다시 동기화. 이동 없는 클릭은 스토어의 no-op 가드가 걸러서 히스토리에 쓰레기 엔트리가 안 남는다 — [PR #25](https://github.com/uulab-official/erd/pull/25).
- [x] Export: SVG / PNG / PDF / Markdown / JSON — [PR #5](https://github.com/uulab-official/erd/pull/5)
- [x] SVG Exporter 시각 품질 재작성 (PNG/PDF는 이 SVG를 그대로 래스터화하므로 셋 다 동시에 개선됨) — PR #5의 자리표시자 그대로 방치돼 있던 걸 캔버스(`EntityNode`/`ErdCanvas`)와 같은 시각 언어로 다시 그렸다: PK/FK 뱃지, 컬럼 타입+not-null `*` 마커, 관계선은 Entity 박스 모서리에서 나가고(중심을 관통하지 않음) 화살표 마커 + `이름 (cardinality)` 라벨 + non-identifying dashed, Subject Area 박스(점선+색상 틴트+이름 칩), Memo(포스트잇 박스+줄바꿈 텍스트)까지 렌더링 — Subject Area/Memo는 그동안 Model에 있어도 export에는 전혀 안 그려지고 있었다. **버그도 하나 잡음**: Subject Area 박스는 멤버 Entity 바깥으로 패딩되면서 좌표가 음수가 될 수 있는데(원점 근처 Entity + Subject Area padding), 기존 코드는 `viewBox="0 0 w h"`로 고정해서 그 박스가 통째로 캔버스 밖에 잘려 안 보였다 — 전체 콘텐츠(Entity/Memo/Subject Area 박스)의 실제 바운딩 박스를 구해 `viewBox`의 원점 자체를 음수로 옮기도록 고쳤다 — [PR #31](https://github.com/uulab-official/erd/pull/31)
- [x] Mermaid Exporter — `packages/sdk`의 `Exporter.targetFormat`에 `"mermaid"`가 처음부터 선언돼 있었지만 구현이 없었다(Import는 되는데 Export는 안 되는 비대칭). `renderMermaid`가 `erDiagram` 문법으로 Entity 블록(타입+이름+PK/FK/UK 마커)과 Relationship 라인(cardinality/optionality를 `||--|{` 류 crow's-foot 표기로 매핑)을 생성. 댕글링 참조(존재하지 않는 Entity를 가리키는 Relationship)는 깨진 문법을 만들지 않도록 건너뜀 — [PR #28](https://github.com/uulab-official/erd/pull/28)
- [x] Appwrite Import — `appwrite.json`(Appwrite CLI export) 파일을 파싱해 Model로 가져오기, `$id` 암묵적 PK 합성 — [PR #9](https://github.com/uulab-official/erd/pull/9)
- [x] DBML Import (dbdiagram.io 호환) — `packages/parser`의 `parseDbml`: Table/alias/quoted 식별자, 컬럼 settings(pk/unique/not null/default/note/inline ref), standalone `Ref:`(`>`/`<`/`-` 모두 정규화해 one→many 방향으로 생성, `-`는 1:1+unique), `indexes { }` 블록(복합 컬럼/unique/name), `Enum` 블록(+enum 타입 컬럼 매핑), `//`·`/* */` 주석. Toolbar Import 버튼이 확장자로 분기(`.json`→Appwrite, `.dbml`→DBML, `.mmd`/`.mermaid`→Mermaid)하고, 텍스트 포맷은 좌표가 없으므로 import 직후 layered 레이아웃을 자동 적용. 텍스트 import는 현재 Model의 id/name/adapter를 유지해 Save가 같은 문서에 계속 쓰인다(Appwrite JSON import는 기존 동작 유지). 미지원 구문(Project/TableGroup/increment 등)은 조용히 건너뛰고, 깨진 Ref는 import 전체를 실패시키지 않고 드롭 — [PR #27](https://github.com/uulab-official/erd/pull/27)
- [x] Appwrite 실시간(live) Collection Import — `functions/deploy-appwrite`에 `{ action: "list" }` 요청을 추가, 실제 배포된 Collection(속성/인덱스 포함, 25개 단위 커서 페이지네이션)을 읽어 `appwrite.json`과 동일한 wire 포맷으로 반환하고 기존 `parseAppwriteJson`/`fromNativeSchema`를 그대로 재사용. Toolbar의 "Import Live" 버튼(`canDeploy`가 true일 때만 노출, 같은 Function id 재사용) — [PR #15](https://github.com/uulab-official/erd/pull/15)
- [x] Appwrite Deploy 실행 — `functions/deploy-appwrite`(Appwrite Function, `node-appwrite`)가 `AppwriteAdminAPI`를 실제 구현. `apps/web`은 `Functions.createExecution()`으로 트리거만 하고, 실제 Databases 관리자 호출은 전부 Function 안에서 실행됨(동적 `APPWRITE_FUNCTION_API_KEY` 사용) — [PR #14](https://github.com/uulab-official/erd/pull/14)
- [x] "프로젝트" 개념 (다중 Model + Dashboard + 즐겨찾기/최근 목록) — `ModelStore`에 `list()`/`remove()` 추가, `apps/web`의 `Dashboard` 컴포넌트가 랜딩 화면이 되어 Model 생성/열기/삭제, Toolbar의 "← Models" 버튼으로 되돌아옴. "최근"은 Appwrite `listDocuments`의 `$updatedAt` 내림차순(서버 데이터)으로, "즐겨찾기"는 로컬 `localStorage`(기기별 UI 취향, 서버 스키마 변경 불필요)로 구현 — [PR #16](https://github.com/uulab-official/erd/pull/16). `Project`(dictionary/domains/namingRules/subjectAreas를 포함하는 전체 컨테이너, `packages/schema-engine`의 `Project` 타입)는 여전히 미구현 — 그건 Phase 2의 Dictionary/Domain/Naming Rules가 각각 구현된 뒤에야 의미가 있는 상위 컨테이너라 이번 스코프에서는 제외.

## Phase 3

- [ ] AI ERD 생성/수정 (`AIProvider` 플러그인 인터페이스는 `packages/sdk`에 정의됨 — 실제 Provider 구현 필요)
- [x] ORM Generator — Prisma(`schema.prisma` 생성: 1:1/1:N `@relation`, M:N implicit array, `@id`/`@unique`/`@default`) — [PR #12](https://github.com/uulab-official/erd/pull/12). Drizzle/TypeORM/JPA/Hibernate/Entity Framework는 아직
- [x] SQL Generator/Adapter — PostgreSQL/MySQL/SQLite 3개 dialect 전부 구현 (`SqlDialect` 공통 팩토리, DDL Export, Deploy Plan) — [PR #10](https://github.com/uulab-official/erd/pull/10), [PR #11](https://github.com/uulab-official/erd/pull/11). SQLite는 `ALTER TABLE`로 FK를 추가할 수 없어 `CREATE TABLE`에 인라인으로 넣는다(`supportsAlterForeignKey: false`) — 이미 배포된 테이블에 FK를 새로 추가/삭제하는 경우는 SQL 없이 "테이블 재생성 필요" 경고만 낸다.
- [x] SQL DDL 정합성 버그 수정 — integer/bigint PK 컬럼의 auto-increment DDL이 세 dialect 어디에도 전혀 생성되지 않고 있었다(`SERIAL`/`AUTO_INCREMENT`/`AUTOINCREMENT` 어느 것도 없이 그냥 `integer NOT NULL PRIMARY KEY`만 나감 — auto-increment로 모델링한 정수 PK가 실제로는 절대 자동 채번되지 않는 DDL을 생성하던 실사용 버그). 단일 컬럼 integer/bigint PK + 명시적 default 없음 조건에서 `SqlColumnDef.autoIncrement`를 계산(`toNativeSchema.ts`)하고, PostgreSQL은 `serial`/`bigserial` 타입 치환, MySQL은 `AUTO_INCREMENT` 접미사, SQLite는 `INTEGER PRIMARY KEY AUTOINCREMENT` 인라인(별도 `PRIMARY KEY (...)` 줄 생략)으로 각 dialect가 다르게 렌더링하도록 `SqlDialectOptions`에 훅 3개 추가 — [PR #35](https://github.com/uulab-official/erd/pull/35). 후속으로 `planSqlDeployment`(`plan.ts`)의 ALTER 경로도 감사 — 이미 배포된 테이블에서 PK/default를 바꿔 auto-increment 여부가 바뀌는 경우, 기존 `ALTER COLUMN ... TYPE` 문은 그 변화를 전혀 표현하지 못하는데도 경고 없이 "적용됨"처럼 보이는 스텝을 냈다. 억지로 정확한 ALTER DDL을 만들어내려 하지 않고(3개 dialect 전부 auto-increment 토글은 컬럼 재생성/시퀀스 조작이 필요해 잘못 추측할 위험이 큼), SQLite FK ALTER 제약과 같은 "추측하지 말고 경고" 정책으로 명시적 경고를 추가. `mapTypeBack`에 `serial`/`bigserial` 역매핑도 방어적으로 추가(현재 호출자는 없지만 향후 라이브 스키마 introspection 연결 시 대비) — [PR #36](https://github.com/uulab-official/erd/pull/36)
- [x] GraphQL SDL Generator — `packages/generator`의 `graphqlGenerator`: Entity→`type`, Attribute→field(PK/FK는 실제 저장 타입과 무관하게 `ID`로 매핑 — Postgraphile/Hasura/Prisma의 GraphQL 레이어가 전부 쓰는 관례), not-null은 `!` 접미사, Relationship→참조 필드(one-to-many는 non-null 리스트, one-to-one은 optional 단일 참조, many-to-many는 스칼라 FK 없이 양쪽 다 리스트 — Prisma 제너레이터와 동일한 관례로 필드명 규칙까지 맞춰서 두 생성물이 "같은 도메인의 다른 언어 표현"으로 읽히게 함), `BigInt`/`DateTime`/`JSON` 커스텀 스칼라는 실제로 쓰일 때만 선언. ## 진행 중 알아두면 좋은 것

- `packages/*`의 엔진 코드(Operation, Adapter, Exporter)는 대부분 UI보다 앞서 구현되어 있다. 새 화면을 만들 때 먼저 해당 엔진에 필요한 기능이 이미 있는지 확인할 것.
- Appwrite `apply()`(실제 배포 실행)와 실시간 Collection 목록 조회는 의도적으로 브라우저에서 직접 호출하지 않는다 — API 키가 필요한 관리자 작업이라 서버(Appwrite Function, `node-appwrite`)에서 실행해야 한다. 정적 `appwrite.json` 기반 Import는 이 제약 없이 이미 동작한다.
- `apply`와 `list`(실시간 Import)는 **같은** `functions/deploy-appwrite` Function을 `action` 필드로 분기해서 쓴다 — `apply`가 이미 write 스코프를 요구하므로 `list`의 read-only 호출을 별도 Function으로 분리해도 권한 범위가 줄지 않고, 배포 보일러플레이트만 두 배가 된다. `list`는 Databases API의 원본 `$id`/`attributes`/`indexes`를 그대로 반환하고(`src/listSchema.ts`), attribute 형태 변환은 하지 않는다 — 호출자(`packages/api`의 `invokeListAppwriteSchema`)가 기존 `parseAppwriteJson`을 그대로 재사용해서 파싱 로직 중복을 피한다.
- Appwrite Import는 컬렉션마다 암묵적 `$id` 기반 PK 속성을 합성한다(`packages/deploy-engine`의 `fromNativeSchema`) — Appwrite의 Attributes API/`appwrite.json`은 시스템 필드인 `$id`를 커스텀 attribute 목록에 포함하지 않기 때문. 이미 `id`라는 이름의 커스텀 attribute가 있으면 합성하지 않는다(이름 충돌 방지).
- `createPostgreSQLAdapter`의 `apply()`도 Appwrite와 동일한 이유로 브라우저에서 직접 실행하지 않는다 — 실제 Postgres 연결(`pg` 등)이 필요한 서버사이드 `SqlExecutor` 구현체를 주입받는 구조다. 현재 apps/web에는 SQL Export만 연결되어 있고 Deploy Plan/apply UI는 Appwrite 전용이다.
- `functions/deploy-appwrite`는 pnpm workspace 멤버가 **아니다** — Appwrite 빌드 컨테이너가 이 폴더만 떼어내 `npm install`을 돌리기 때문에 `workspace:*` 의존성을 해석할 수 없다. `src/types.ts`에 `packages/sdk`/`packages/deploy-engine`의 관련 타입을 손으로 복사해 동기화한다 (주석에 명시).
- 이 Function은 정적 API 키가 아니라 Appwrite가 실행마다 주입하는 동적 `APPWRITE_FUNCTION_API_KEY`를 사용한다 (Function의 Execute/Scopes 설정에 따라 스코프가 제한됨). 대화 중 공유된 정적 API 키는 `apps/web/.env.local`의 `APPWRITE_API_KEY`에 보관용으로만 있고 어떤 코드도 읽지 않는다 — 채팅에 노출된 적이 있다면 Console에서 재발급 권장.
- `modelsTableId` 컬렉션에 `name`(string) attribute를 추가해두면 Dashboard의 `list()`가 각 Model의 전체 `data` JSON을 파싱하지 않고도 이름을 보여줄 수 있다 — 필수는 아니다(그 attribute가 없는 기존 row는 `data`를 파싱해 이름을 채우는 fallback이 `packages/api`의 `createAppwriteModelStore`에 있음), 있으면 더 가볍다.
- Model 목록에 사용자별 소유권/권한 분리(현재는 프로젝트 내 모든 Model이 공유되어 보임)는 아직 없다 — 이 앱의 다른 어떤 기능(Import/Deploy 등)도 아직 사용자별로 스코프되어 있지 않아서, 이번 Dashboard도 기존 보안 모델을 그대로 따른다. 실제 멀티테넌시가 필요해지면 Appwrite의 document-level permission(`Permission.read(Role.user(...))`)을 도입해야 한다.
- `Domain`/`Dictionary`/`NamingRuleSet`은 `Model`에 붙는다(원래 `docs/schema-engine.md` 스펙 초안은 이걸 `Project`에 붙였음) — `Project`가 실제 영속화 계층이 없는 지금은 Model 단위가 더 단순하다. 여러 Model이 governance를 공유해야 하는 시나리오가 생기면 그때 끌어올린다.
- `AssignDomain`/`UnassignDomain`의 inverse는 항상 "대상 Attribute의 정확한 이전 상태(domainId+type+length+scale)를 담은 `UnassignDomain`"이지, "그 Domain으로 다시 assign"이 아니다 — `updateDomainCascade`/`deleteDomainCascade`처럼 Domain 자체와 그걸 참조하는 Attribute들을 같은 Transaction에서 함께 undo할 때, inverse가 Domain을 다시 조회해서 값을 가져오면 undo 순서에 따라 Domain이 아직 되돌려지지 않은 상태를 읽어버리는 버그가 생긴다 (실제로 겪음 — `packages/erd-engine/src/operations/attribute.ts`의 `priorDomainState` 주석 참고).
- ~~attribute 편집 UI는 없다~~ → 이제 있다: `apps/web`의 `EntityInspector`(엔티티 클릭 → 우측 사이드패널)에서 로지컬/피지컬 이름, attribute 추가/삭제/이름/타입/PK·FK·Unique·Nullable 플래그, Domain 배정까지 전부 편집 가능 — [PR #22](https://github.com/uulab-official/erd/pull/22). GovernancePanel의 "entity.attribute" 플랫 목록 방식 Domain 배정도 그대로 남아있다(둘 다 결국 같은 `assignDomain`/`unassignDomain` 액션을 호출).
- Version 스냅샷은 Model 문서의 새 `versions`(JSON 배열) attribute에 저장된다 — `name`과 마찬가지로 없어도(오래된 row) 정상 동작하고(빈 배열로 취급), 있으면 그대로 쓴다. 매 `saveVersion`/`deleteVersion` 호출이 문서를 전체 읽고 배열을 수정해 다시 쓰는 read-modify-write라, Appwrite `updateDocument`가 부분 업데이트를 지원해도(전체 document를 다시 보낼 필요 없음) 동시에 같은 Model에 여러 명이 버전을 저장하면 마지막 쓰기가 이길 수 있다 — 실시간 협업(Phase 3)이 붙기 전까지는 감내할 수 있는 수준.
- Restore는 복원한 스냅샷을 `savedModel`(= Diff 탭의 baseline)로도 취급한다 — `importModel`과 같은 패턴. 복원 직후엔 Diff 탭이 "변경 없음"으로 깨끗하게 보이고, 그 상태를 실제로 Appwrite에 반영하려면 여전히 "Save"를 눌러야 한다(Restore 자체는 메모리상의 현재 모델만 바꾼다).
- `apps/web`/`packages/ui`에 실제 디자인 시스템이 생겼다 — `packages/ui`의 `Button`/`Input`/`Select`/`Badge`/`Card`/`Tabs`가 전부 자체적으로 완결된 Tailwind 유틸리티 클래스를 갖는다(다른 패키지가 어딘가에 매칭되는 CSS를 정의해줄 거라고 가정하지 않음 — `Button`이 `mf-button--*` 클래스만 내보내고 그 클래스를 실제로 스타일링하는 CSS가 저장소 어디에도 없었던 버그를 고치면서 확립한 원칙). 색상 토큰(`brand` indigo 스케일)과 폰트(Inter, Google Fonts)는 `apps/web/tailwind.config.js`/`index.html`에 있다 — `packages/ui`는 자체 Tailwind 설정이 없고 소비하는 앱의 빌드에 얹혀가므로, 새 앱이 이 패키지를 쓰려면 그 앱의 `tailwind.config.js`에도 `brand` 색상을 정의해야 정확히 같은 룩이 나온다.
- React Flow 캔버스(`packages/canvas`)의 노드/엣지는 이번 디자인 패스에서 건드리지 않았다 — 여전히 기본 검은 테두리 박스다. 커스텀 노드/엣지 컴포넌트는 ROADMAP Phase 1의 "Canvas Engine" 항목(아직 없음)에서 다룰 별도 작업이다.
- Relationship Inspector에서 "Delete Relationship"은 Relationship만 지운다 — `connectEntitiesCascade`가 만들어줬던 FK Attribute는 남는다(데이터를 담고 있거나 다른 곳에서 참조될 수 있어 자동 삭제가 더 위험). 남은 FK는 Validation이 경고로 잡아주고, Entity Inspector에서 직접 지우면 된다. 캔버스의 `onSelectEntity`/`onSelectRelationship` 선택은 `useSelectionStore`에서 상호배타적으로 관리된다(pane 클릭은 둘 다 null로 보고하므로, relationship의 null이 entity 선택을 지우지 않게 방어 로직 있음).
- Index 관리(`CreateIndex`/`DeleteIndex` Operation + `EntityInspector`의 "Indexes" 섹션)가 추가됐다 — `docs/operations.md`에 스펙만 있고 구현이 없던 항목을 채운 것. erwin/erdcloud와 동일하게 `UpdateIndex`는 없다 — 이름/컬럼/unique를 바꾸려면 삭제 후 재생성한다(다른 Operation들과 달리 부분 수정이 필요할 만큼 필드가 많지 않아서 별도 Operation을 만들 이유가 없었다). Index의 `id`는 `crypto.randomUUID()`로 생성하고, 기본 이름은 `idx_{physicalName}_{attribute names}` 패턴(attribute id가 아니라 name을 이어붙임 — id는 ULID라 이름에 넣으면 읽을 수 없다) — [PR #23](https://github.com/uulab-official/erd/pull/23).
